# Data Directory Guide

## What Each Top-Level Folder Means

- `configs/`
  - Central runtime configuration (paths, model/tokenizer IDs).
  - `workflow_paths.json` is used by notebook orchestration and scripts.

- `data/raw/`
  - Source-of-truth dataset files from COCO download.
  - Should remain unchanged after acquisition.

- `data/processed/`
  - Generated artifacts from pipeline stages (alignment, tokenized data, manifests, CLIP features).

- `artifacts/`
  - Saved model artifacts/checkpoints from training stages.

- `src/`
  - Source-of-truth implementation code.
  - Notebook should call these modules, not duplicate logic.

- `notebooks/`
  - Storytelling/orchestration layer for running stages in order.

- `notes/`
  - Team-facing operational documentation and run guides.

- `scripts/`
  - Utility scripts (for example environment checks).

## Overview

`data/` is organized by lifecycle:

- `data/raw/`: original downloaded dataset files (do not modify)
- `data/processed/`: pipeline-generated intermediate files used for training

## `data/raw/`

Expected contents:

- `train2017/`: COCO training images
- `annotations/`: COCO annotation JSON files
- optional archives like `train2017.zip` and `annotations_trainval2017.zip`

Purpose:

- Source-of-truth input data from COCO download/acquisition step.
- Should remain unchanged once downloaded.

### What COCO files are in `raw/`?

COCO is not only images. You also download annotation metadata files.

- `train2017/`
  - JPEG image files (the actual visual inputs).
  - Filenames look like `000000203564.jpg`.

- `annotations/captions_train2017.json`
  - Caption supervision used by our Stage 1 prep.
  - Key parts:
    - `images`: image metadata (`id`, `file_name`, width, height)
    - `annotations`: caption entries (`image_id`, `caption`, `id`)
    - `info`, `licenses`: dataset metadata

- Other annotation JSONs may also exist from `annotations_trainval2017.zip`
  - Examples:
    - `captions_val2017.json`
    - `instances_train2017.json`, `instances_val2017.json`
    - `person_keypoints_train2017.json`, `person_keypoints_val2017.json`
  - These are for detection/keypoints tasks and are not required by our current Group 1 baseline path.

### `raw/` File-by-File Meaning

- `data/raw/train2017/*.jpg`
  - Original COCO training images (actual visual inputs).

- `data/raw/annotations/captions_train2017.json`
  - Main caption supervision used by Group 1 baseline.
  - Read by `prepare_stage1_dataset.py` to build `alignment.json`.

- `data/raw/annotations/captions_val2017.json`
  - Validation caption annotations (not used in current baseline training flow).

- `data/raw/annotations/instances_train2017.json`, `instances_val2017.json`
  - Object detection annotations (boxes/categories), not used in this baseline path.

- `data/raw/annotations/person_keypoints_train2017.json`, `person_keypoints_val2017.json`
  - Human keypoint annotations, not used in this baseline path.

- `data/raw/train2017.zip`, `data/raw/annotations_trainval2017.zip`
  - Download archives kept for reuse/re-extract; optional to keep after extraction.

## `data/processed/`

Expected subdirectories:

- `stage1_alignment/`
  - `alignment.json`: image-caption rows built from COCO annotations
  - `alignment_chat.json`: instruction-response format for stage 1
  - `alignment_tokenized.json`: tokenized stage-1 samples
  - `stage1_manifest.json`: stage-1 loader manifest (text + vision feature path)

- `stage2_finetuning/`
  - stage-2 input/alignment rows
  - `alignment_tokenized_stage2.json`: tokenized stage-2 samples
  - `stage2_manifest.json`: stage-2 loader manifest

- `clip_embeddings/`
  - precomputed CLIP feature files (`.npy`) per image

Purpose:

- Stores all derived artifacts generated by the workflow stages.
- Safe to regenerate intentionally when rerunning pipeline steps.

### `processed/` File-by-File Meaning

- `data/processed/stage1_alignment/alignment.json`
  - Flattened image-caption rows built from COCO captions.

- `data/processed/stage1_alignment/alignment_chat.json`
  - Instruction-response version of `alignment.json` for Stage 1.

- `data/processed/stage1_alignment/alignment_tokenized.json`
  - Stage 1 tokenized text (`input_ids`, `labels`) plus image filename.

- `data/processed/stage1_alignment/stage1_manifest.json`
  - Loader-ready rows mapping tokenized text to `vision_path` (`.npy` CLIP features).

- `data/processed/stage2_finetuning/alignment_tokenized_stage2.json`
  - Stage 2 tokenized text (typically longer max length, Group 1-compatible fallback may reuse Stage 1 chat rows).

- `data/processed/stage2_finetuning/stage2_manifest.json`
  - Stage 2 loader-ready mapping (same structure as Stage 1 manifest).

- `data/processed/clip_embeddings/*.npy`
  - Precomputed CLIP visual features per image.
  - Training reads these features directly to avoid recomputing CLIP each step.

## Output Files You Will Commonly See

After running Stage 2 and Stage 4, it is normal to have:

- `data/processed/stage1_alignment/alignment_tokenized.json`
- `data/processed/stage2_finetuning/alignment_tokenized_stage2.json`
- `data/processed/stage1_alignment/stage1_manifest.json`
- `data/processed/stage2_finetuning/stage2_manifest.json`

And from Stage 3:

- `data/processed/clip_embeddings/*.npy`

## Why Stage 1 and Stage 2 Manifest Row Counts Can Match

If Stage 2 falls back to Stage 1 chat data (Group 1 reproduction behavior),
then Stage 1 and Stage 2 tokenized datasets come from the same source rows.
In that case it is expected that:

- `stage1_manifest.json` row count == `stage2_manifest.json` row count

The difference is in tokenization settings (for example Stage 2 typically uses
a longer max sequence length), not in number of rows.
